# IKIO TDS Generator — What Comes Automatically vs. What You Do

A field-by-field guide to the Spec Sheet Editor. It explains **everything the system fills in
for you automatically** the moment an extraction finishes, and **what you are expected to do**
in the form on the left before exporting the final IKIO TDS.

> TL;DR — Upload a vendor PDF and the app produces a *complete, ready-to-review* 4-page IKIO
> template with the product name, category, overview specs, description, features, certifications,
> qualification badges, spec tables, ordering codes, accessories and dimensions **already filled
> in**. Your job is mainly to **add/crop the product & dimension images**, fill the **header/project
> fields**, and **verify every value against the PDF** (the colored dots tell you what to check).

---

## 1. How it works end-to-end (the pipeline)

When you drop a PDF on the Home screen, this happens automatically:

1. **Page images rendered** — every page is rendered to an image (pypdfium2, 3× scale, JPEG q95)
   and shown in the right-hand **Vendor PDF Review** panel.
2. **Text extracted** — the PDF text layer (pdfplumber) + tables are pulled per page. For
   image-only/scanned pages (< 120 characters, < 3 table rows) **PaddleOCR** runs as a fallback.
   This combined text is the `sourceText` used for the confidence dots.
3. **AI extraction (pass 1)** — an LLM (Gemini by default: `gemini-flash-latest`) reads the text and
   returns a structured IKIO-style JSON draft.
4. **AI review (pass 2)** — a second "reviewer" LLM pass corrects the first (fixes SKU codes leaking
   into wattage/lumens, mismatched variant counts, unit conversions) and is merged in.
5. **Post-processing** — units are forced to **US standard** (inches, °F, lbs), a fixed 40-row spec
   template is applied, ordering columns are normalized, values canonicalized.
6. **Result cached** — keyed by the PDF's hash, so re-uploading the same file is instant and uses no
   AI quota.
7. **Editor draft assembled** — the frontend turns that JSON into the editable draft, and layers on
   its own automatic derivations (name recommendations, fixture prediction, description synthesis,
   badge inference, estimated values, etc. — see below).

## 2. PART A — Fully automatic (already done, no action required)

These are populated for you before you click anything. You can still edit them all.

### Identity & classification
- **Product name (title)** — a unique, IKIO-style name is generated (e.g. *"Orion High Bay Light"*).
  Names come from a star/constellation codename pool and are **reserved globally on the server** so
  no two products ever get the same name. The first recommendation is used as the title.
- **Recommended product names** — a short list of alternates shown as clickable buttons in the
  **Product** section (click one to switch the title).
- **SKU No.** — taken from the vendor's *Part Number / SKU / Catalog Number / Cat Number / Model*.
- **Category** (e.g. Indoor / Outdoor / Hazardous Location Lighting) — predicted from the product name.
- **Sub-category** (e.g. Panel, Linear High Bay, Wall Pack) — predicted from the product name.
- **Fixture group** (High Bays, Panels, etc.) — predicted, shown on the sheet header.
- **Subtitle** — vendor alternate name → product type → "Technical Data Sheet".

> Category, sub-category and group are chosen by matching the product name against a 22-entry fixture
> table (most-specific wins, e.g. *"linear high bay"* beats *"high bay"*). The **product name decides
> first**, so a clean IKIO name overrides messy vendor text.

### Overview table (page 1)
- **Overview rows are auto-built** using a category-appropriate template (indoor / outdoor / high bay /
  low bay / downlight / flood / panel …), then filled from the extracted specs and variant matrix.
- **Every extra vendor spec** not already in the template is appended as its own row (you can hide any).
- **Estimated values** are computed when a value is missing but derivable, and tagged **"(Estimated)"**:
  - Current = Watts ÷ Volts
  - Efficacy = Lumens ÷ Watts
  - Lumen output = Watts × Efficacy
  - Power = Lumens ÷ Efficacy
- **Source-check dots** appear on each row (see §5) so you know what to verify.

### Copy (page 1)
- **Description** — a single ~400–450 character paragraph.
  - If the vendor PDF already has a description, it's used (trimmed).
  - If not, one is **synthesized** from the product type, applications, wattage/lumens/CCT, voltage,
    efficacy, driver, housing and mounting.
  - On a **fresh** extraction the description is also **auto-generated by the AI** ~1.2 s after load —
    you don't need to press Generate.
- **Features** — the vendor's feature list, cleaned (bullets/periods stripped, de-duplicated).
  The extractor targets **exactly 4 benefit sentences (~100 chars each)**.
- **Application areas** — from the extraction, and also auto-filled from the predicted fixture profile
  when empty.
- **Certifications** — assembled from Certifications + IP rating + IK rating + warranty (first 4).
- **Warranty label & copy** — label normalized to 5/7/10 Years; a full warranty paragraph is generated
  to match (default 5-year if unspecified).

### Qualification icons (badges)
- **Auto-selected** by scanning certifications, dimming, IP rating, product name, features and notes
  for keywords: **UL Listed, DLC / DLC Premium, ETL, Energy Star, RoHS, Wet Location (IP65/66),
  Dimmable (0-10V), Sensors, Power Selectable, CCT Selectable**.
- Selectable-power (`|` in the Power value) auto-adds **Power Selectable**; selectable CCT auto-adds
  **CCT Selectable**.

### Additional pages (2–4)
- **Product Specifications (page 2)** — spec groups are built from the **variant matrix**: one group
  per variant with part number, power/lumen options, and shared voltage / CRI / current / THD / light
  distribution / efficacy / CCT. Leaked catalog codes are stripped from power/lumen values.
- **Ordering Information (page 3)** — the decoder is mapped onto 11 fixed columns (Brand, Family,
  Size, Power, Voltage, Dimming, CCT, Distribution, Driver, Finish, Manufacturer). A **Build Order
  Code** helper auto-fills single-option columns and lets you pick multi-option ones.
- **Ordering example** — assembled example part number (from the extraction).
- **Accessories (page 3)** — code + description rows from the extracted accessories list.
- **Dimensions (page 4)** — label + width/height/depth rows from the extracted dimensions (in inches).

### Reference material & boilerplate
- **Vendor PDF pages** — all rendered and shown on the right for cross-checking.
- **Specs note**, **ordering note** — standard IKIO boilerplate pre-filled.
- **IKIO header/footer template** — applied to every page.

## 3. PART B — Auto-filled, but you should VERIFY or CHOOSE

The system fills these, but a human decision improves them:

- **Category / Sub-category** — predicted and editable. Clearing a field makes it **re-predict**; a
  value you type by hand is **never overwritten**. Verify they read correctly.
- **Recommended product name** — the first is applied; pick a different alternate if you prefer.
- **Overview values with amber/red dots** — amber = only partly found in the PDF, red = not found
  (often an *(Estimated)* or AI-inferred value). **Check these against the PDF.**
- **Build Order Code — multi-option columns** — columns with more than one code show a dropdown
  ("Select…"); single-option columns auto-fill and are marked *(auto)*. Choose the options you want,
  then optionally "Use as example".
- **Product image selection** — if multiple images exist (crops/uploads), pick which one is the hero.
- **Warranty length** — defaults to 5 years when the PDF is silent; set 7 or 10 if needed (the copy
  regenerates to match, unless you've hand-edited it).

## 4. PART C — Manual (what you do in the left-hand form)

These are **not** auto-filled — they're yours to enter/curate:

### Header (top of the form)
- **Project Name**
- **CAT.#**
- **Fixture Schedule**
- **Note** (header note)
- **Footer note**

### Images (the main manual effort)
- **Product image** — **Upload** from your computer, or **Crop** it from a vendor PDF page (the app
  intentionally does **not** auto-extract embedded PDF images — you crop the visual you want).
- **Accessory images** — per accessory: Crop / Upload / Edit.
- **Dimension drawings** — per drawing: Crop / Upload / Edit.
- **Image editor** (the *Edit* button) — crop, erase, pen, text, background removal, brightness/
  contrast/saturation, **plus AI Edit** (see §6).

### Review & curate (recommended pass before export)
- **Verify every overview value** against the vendor PDF — trust the green dots, scrutinize amber/red.
- **Reorder overview rows** (drag) and **hide** the ones that shouldn't appear on the sheet.
- **Add/remove** spec groups, power/lumen options, ordering codes, accessories, dimensions as needed.
- **Fill blank spec fields** the extractor left as "Not Specified".
- **Tune the description / features** (edit directly, or use the AI assistant with a custom instruction).
- **Confirm the SKU, part numbers and order codes** — these carry no confidence dots.

### Export
- **Download PDF** — flattened, pixel-accurate PDF of all pages.
- **Editable PDF (Print)** — opens the browser print dialog producing a *true text* PDF with
  selectable/editable text.
- **Reset** — rebuilds the entire draft from the extraction (re-runs name & fixture prediction).

## 5. Reading the source-check dots (confidence)

Each overview value is checked against the vendor PDF text (`sourceText`) and gets a colored dot:

| Dot | Meaning | What to do |
|-----|---------|-----------|
| 🟢 Green | **Found** — every part of the value appears in the PDF | Trust it |
| 🟡 Amber | **Partial** — some parts matched, some didn't | Spot-check against the PDF |
| 🔴 Red | **Not in PDF** — no match (often estimated / AI-inferred) | **Verify manually** |

- Numbers are matched exactly against numbers in the PDF; words match as substrings (≥3 letters).
- **Hover a dot** to see the matched source snippet.
- Dots only appear when the PDF text was captured, and only on values that are actually filled.

---

## 6. AI helpers (on demand)

- **Description / Features assistant** (in the Copy section) — regenerate or rewrite with a custom
  instruction; uses the product name, category, specs and current text as context.
- **AI Edit Image** (in the image editor) — Gemini applies a single requested change to an image
  (e.g. convert *mm → inch* on a dimension drawing, fix a label). Returns a new image; quota-limited.
- **AI annotate** — a cheaper, quota-free path that locates label text and overlays replacements on a
  canvas instead of regenerating the image.

## 7. Good to know / caveats

- **Nothing is invented for empty fields** — missing scalar specs come back as *"Not Specified"*;
  features are **not** padded with fabricated text (only the synthesized *description* has a length
  floor).
- **Units are forced to US standard** by the extractor: inches, °F, lbs. Dimension conversion is the
  highest-priority rule.
- **Re-uploading the same PDF is cached** (instant, no AI cost). Use the vendor panel's re-extract /
  cache purge if you changed the source.
- **Product names are globally unique** — reserved on the server so two different products never
  collide.
- **Auto-prediction respects your edits** — once you type a category/sub-category (or open a saved
  draft), the app won't overwrite it; clearing a field lets it re-predict.
- **Drafts are saved locally** (IndexedDB) as you edit and restored when you reopen a spec.
